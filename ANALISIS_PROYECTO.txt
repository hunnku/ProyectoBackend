A continuación se detallan los requisitos que no se cumplen actualmente en el proyecto y las acciones recomendadas para completarlos.

### 1. Configuración Inicial

*   **Estado:** Parcialmente completado.
*   **Comentarios:** Tienes una buena estructura de carpetas y un servicio web básico con Flask. Sin embargo, faltan dependencias clave en tu archivo `requirements.txt`.
*   **Acciones:**
    1.  **Añadir dependencias de desarrollo:** Agrega las herramientas para pruebas, cobertura y calidad de código a tu `requirements.txt`.
        ```
        # requirements.txt
        
        # Dependencias de la aplicación
        Flask==3.1.3
        # ... (otras dependencias que ya tienes) ...

        # Dependencias de desarrollo
        pytest
        coverage
        pylint
        ```
    2.  **Instalar dependencias:** No olvides instalar las nuevas librerías en tu entorno virtual con `pip install -r requirements.txt`.

### 2. Implementación de Pruebas

*   **Estado:** No iniciado.
*   **Comentarios:** La carpeta `/test` está vacía. Este es un punto crítico a desarrollar.
*   **Acciones:**
    1.  **Crear Pruebas Unitarias (Mínimo 10):** Debes crear archivos de prueba (ej. `test/test_app.py`) y escribir pruebas para las funciones y lógica de negocio en `src/app.py`. Usa `pytest` para esto.
        *   **Ejemplo de prueba unitaria:** Probar la lógica de creación de un usuario.
    2.  **Crear Pruebas de Integración (Mínimo 5):** En el mismo archivo de pruebas, puedes probar los endpoints de tu API para asegurar que responden correctamente.
        *   **Ejemplo de prueba de integración:** Realizar una petición `GET` al endpoint `/api/usuarios` y verificar que el código de estado sea 200 y que la respuesta sea un JSON válido.
    3.  **Aplicar TDD (Test-Driven Development):** Para dos nuevas funcionalidades, escribe primero las pruebas que fallen y luego el código que las haga pasar. Por ejemplo, podrías añadir un endpoint para archivar tareas en lugar de eliminarlas.
    4.  **Configurar Cobertura de Código:**
        *   Ejecuta tus pruebas con el comando: `coverage run -m pytest`.
        *   Genera un informe en la consola con: `coverage report`.
        *   Para configurar el umbral del 80%, crea un archivo `.coveragerc` en la raíz de tu proyecto.
            ```ini
            ; .coveragerc
            [report]
            fail_under = 80
            show_missing = True
            
            [run]
            branch = True
            ```

### 3. Pipeline CI/CD con GitHub Actions

*   **Estado:** No iniciado.
*   **Comentarios:** No existe un flujo de trabajo de GitHub Actions.
*   **Acciones:**
    1.  **Crear el workflow:** Crea la ruta de carpetas `.github/workflows/` en la raíz de tu proyecto.
    2.  **Definir el archivo `ci.yml`:** Dentro de `workflows`, crea un archivo llamado `ci.yml` con los siguientes stages:
        ```yaml
        # .github/workflows/ci.yml
        name: CI Pipeline
        
        on: [push, pull_request]
        
        jobs:
          build:
            runs-on: ubuntu-latest
            
            steps:
            - name: Checkout code
              uses: actions/checkout@v3
              
            - name: Set up Python
              uses: actions/setup-python@v4
              with:
                python-version: '3.10' 
            
            - name: Install dependencies
              run: |
                python -m pip install --upgrade pip
                pip install -r requirements.txt
                
            - name: Lint with Pylint
              run: |
                pylint src/ --fail-under=8.0
                
            - name: Test with Pytest and generate coverage
              run: |
                coverage run -m pytest
                coverage report --fail-under=80
                coverage html
            
            - name: Archive code coverage results
              uses: actions/upload-artifact@v3
              with:
                name: coverage-report
                path: htmlcov
        ```
    3. **Análisis Estático con SonarCloud:** Esto requiere configuración adicional. Deberás crear una cuenta en SonarCloud, vincularla a tu repositorio de GitHub y añadir los secrets `SONAR_TOKEN` a tu repositorio para que el workflow de Actions pueda enviar el análisis.
